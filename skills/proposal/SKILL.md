## Шаг 1 — Сбор исходных данных

Перед генерацией КП собери у пользователя следующее. Если часть данных уже есть в контексте разговора — используй их, не переспрашивай.

**Обязательно:**

**Желательно:**

- Дополнительный контекст из переговоров (что обсуждали на встрече)

Если минимальных данных нет — задай 2-3 уточняющих вопроса перед генерацией.

## Шаг 3 — Структура КП

Генерируй КП строго по этой структуре. Объём: 1.5–2 страницы A4.

### Блок 1: Заголовок и адресат
```
Коммерческое предложение
Для: [Имя, Должность, Компания]
От: Илья Болховский, эксперт по ИИ-автоматизации
Дата: [текущая дата]
```

### Блок 2: Задача клиента (2–4 предложения)
Перефразируй боль клиента его языком. Покажи, что понял проблему.
Пример: «[Компания] обрабатывает X заявок в неделю. Менеджеры вручную идентифицируют позиции в каталоге из 15 000 наименований. На одну заявку уходит 30–60 минут — это Z часов команды в день.»

### Блок 3: Решение
Название продукта + 4–6 пунктов что входит. Используй данные из `references/products.md`.

### Блок 4: Результат (ROI-блок) ← САМЫЙ ВАЖНЫЙ
Конкретные цифры. Формат:

```
До:     [текущая ситуация в цифрах]
После:  [результат после внедрения]
Выгода: [экономия/доход в рублях или часах/месяц]
```

Используй данные из `references/roi-arguments.md`. Если у клиента есть конкретные данные — считай индивидуально.

### Блок 5: Как начать — Design Sprint
Предлагать всегда при чеке >150K или при наличии технической неопределённости.

**Ключевое правило подачи:** продавать не «процесс», а конкретный артефакт.

❌ Не так: «За 7 дней проведём анализ и подготовим отчёт»
✅ Так: «Через 7 дней у вас на руках будет работающий прототип — [конкретное описание что делает прототип на данных клиента]. Ваши [менеджеры/эксперты] смогут его протестировать и решить: работает или нет. Это не презентация. Это инструмент.»

Шаблон текста:
```
Предлагаю начать с Design Sprint ([X] дней, [цена] руб):
— [Конкретный артефакт на данных клиента]
— Протестирован на [N] реальных [запросах/документах/кейсах]
— [Что клиент сможет оценить по итогу]
Оплата — после сдачи результатов.
Стоимость засчитывается в счёт основного проекта.
```

**Фрейм страховки** (если клиент колеблется):
«Альтернатива — сразу в полную разработку за [500K]. Риск: через 2 месяца выяснится, что [специфика данных клиента] требует кастомной доработки. Design Sprint — страховка от этого. Дешёвая.»

### Блок 6: Стоимость и сроки
```
Стоимость: [сумма] руб
Срок:      [недели]
Оплата:    50% предоплата, 50% по факту сдачи
           (для Design Sprint: 100% после сдачи результатов)
```

### Блок 6.5: Инфраструктурные расходы (включать если проект >150K)
```
После запуска — ежемесячные расходы на инфраструктуру:
— [Платформа]: ~[X] руб/мес
— API (ИИ-модели): ~[Y] руб/мес
— Сервер (VPS): ~[Z] руб/мес
Итого: ~[сумма] руб/мес
```

Типовые значения: см. `references/products.md` (секция «Инфраструктура»).

### Блок 7: Почему я
3–4 коротких довода. Используй данные из `references/trust-signals.md`.
Выбирай аргументы релевантные должности ЛПР.

### Блок 8: Следующий шаг
Конкретный призыв к действию. Один, чёткий.
Пример: «Ответьте на это письмо или напишите в Telegram @bolkhovsky — договоримся на 30-минутный звонок, чтобы обсудить детали.»

## Критические правила тона

**ВСЕГДА:**
- Конкретные цифры вместо «значительно» и «существенно»
- «Сокращаю время с X до Y» вместо «оптимизирую процессы»
- ROI в рублях или часах — в каждом КП
- Российский стек: Bitrix24, SaluteSpeech, Nodul, 152-ФЗ
- Design Sprint = конкретный артефакт, не «анализ и отчёт»

**НИКОГДА:**
- «Инновационные технологии», «экосистема ИИ», «цифровая трансформация»
- Абстрактные обещания без цифр
- Перечисление технологий без связи с бизнес-результатом
- Слово «синергия»

## Проверка перед выдачей

- [ ] Есть имя и компания адресата
- [ ] ROI-блок содержит реальные цифры (не «до 50%»)
- [ ] Design Sprint описан через артефакт, а не через процесс
